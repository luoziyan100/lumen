# Brief:项目(Projects)——多会话共享工作区,作为独立功能

> 状态:方案定稿,待 owner 排期 · 2026-07-05
> 触发:owner 观察「工作区资料全局共有,会话间互相干扰」;讨论收敛为按项目分家,
> 并明确要求:**作为独立功能单独做,不与现有单会话体验混淆**。

## 定位(一句话)

同一课题下的多个会话共享同一个工作区(资料+产物);不同项目物理隔离。
对标 ChatGPT / Claude 的 Projects,用户已有现成心智模型。

## 边界(owner 的「别混淆」三原则,违者打回)

1. **默认体验零变化**:不建项目的用户,一切与今天完全相同(现有数据自动归入隐形的
   default 项目,无迁移、无感知)。
2. **共享的只是工作区文件(与后续的项目指令),永不共享聊天记录**:会话线程始终是
   白纸起步,文件是拉取式——这是架构既有事实,产品层面也如此承诺。
3. **与记忆开关解耦**:会话级「参考资料库 开/关」(开卷/闭卷)是另一个独立小功能,
   连同"引用旧产物的来源标注",单独排期,不捆绑在本 brief。

## UI 形态

- 标题栏品牌名右侧:项目切换器(Kumo Select/DropdownMenu)
  `[默认 ▾] → 默认 | 预测编码 | ＋ 新建项目…`
- 切换项目 = 侧栏会话列表、工作区抽屉、新会话归属整体切换;最近项目记 localStorage。
- 项目重命名/归档/删除与「项目指令」(注入系统提示的项目级说明)放阶段 2。

## 落地清单

**服务端(小)**:protocol 增 `list_projects` / `create_project` 两个消息——task 表、
工作区目录(`~/.lumen/workspaces/<project_id>/`)、全部现有接口本就按 projectId 参数化,
只缺枚举与创建入口。阶段 2 增 project instructions 的存储与注入。

**前端(中)**:App 的 `PROJECT` 从写死常量改为状态(localStorage 记最近项目);
切换器组件;`useAgent` / `useWorkspace` / `client.*` 已带 projectId 形参,透传即可。

## 验收

1. 两个项目互不可见彼此的会话与资料(list / list_assets / 上传归属全部按当前项目);
2. 不建项目的用户全程无感知,旧数据完好在 default 项目;
3. 切换项目后新建会话/上传文件落入正确项目(服务端目录与 task 表核对);
4. 照 §9 纪律:协议契约测试覆盖两个新消息;e2e 用真实路径跑一遍双项目隔离。
