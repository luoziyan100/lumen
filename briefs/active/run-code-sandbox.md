# Brief:run_code 工具与本地沙箱(通用 agent 线)

> 状态:方案已定稿,待 owner 拍板排期 · 2026-07-05
> 触发:产品拍板「主形态 = 桌面本地应用」+「通用 agent」方向后,owner 提问:
> 让模型跑代码任务,是否需要沙箱、是否要给它一个 node/执行环境。

## 结论(先行)

**要安排,但分两档:**

- **Lumen 研究版:维持 §5.4 立场不变**——不给 bash/run_code。研究场景 fs 原语 + 研究桥接已覆盖,
  不为通用性给研究版开攻击面。
- **通用 agent 线(Lumen 2.x / Prism 后端)**:`run_code` 进路线图,**设计即带沙箱**,按下述三层做。

## 威胁模型(本地单用户 ≠ 云多租户)

云沙箱防"租户逃逸伤平台";本地 app 防的是**模型不受控地伤到用户自己**:误删文件、读隐私目录外传、
写持久化(LaunchAgent/crontab)。因此目标是 **限权 + 可见 + 可撤销**,不追求 VM 级隔离(那是云版的事)。

## 方案:三层,取前两层

**L1 进程纪律(必做,零新依赖)**
- `cwd` 锁定沙箱工作区;超时(默认 60s)+ 输出上限(截断存工作区文件,线程给路径);
- 接 AbortSignal(全工具既有约定);
- **命令、退出码、输出全量进 task_events**——「诚实可见」铁律的自然延伸,UI 过程行可展开审计。

**L2 OS 级限权(必做,macOS)**
- `sandbox-exec` + Seatbelt profile 模板:文件系统只允许读写工作区(+ 只读系统运行时);
  **网络默认全禁**,按任务放行(localhost / 域白名单);禁提权。
- deprecated 但 macOS 自身与主流本地工具仍在用,单机产品足够;Linux 自托管对应 bubblewrap/landlock,
  接口留成策略注入(SandboxPolicy 端口),平台各自实现。

**L3 容器/microVM(不做,记录理由)**
- Docker/OrbStack = 要求用户装重依赖,桌面产品不可接受;WASM/WASI 跑不动 node 生态。
- 云托管多租户版必须上容器级隔离——到那天再做,本地版不背这个复杂度。

## 执行环境从哪来(回答"给它一个 node")

**M7.1 要 bundle 进 app 的那颗 node,就是执行环境**——跑 agent-service 与跑用户代码共用,一石二鸟,
不额外分发 runtime。python 探测系统解释器(python3 -V),没有就让工具如实报"本机未检测到 python"。

## 权限模型(产品面,开源卖点)

- 默认:工作区内执行,静默放行(有 L2 兜底);
- 越权动作(联网 / 装依赖 / 触碰工作区外路径)→ 过程行内**明确请求授权**,同意后本任务内生效;
- 所有执行历史可回看(事件流)。权限透明是对 BYOK/自托管人群的核心信任承诺。

## 落地位置与验收

- 新工具 `agent-service/src/tools/env/run-code.ts`(Tool 契约 / Workspace 端口 / AbortSignal 全部现成);
  `SandboxPolicy` 端口 + macOS Seatbelt 实现;arch doc §5.4 随实施更新立场。
- 验收(照 §9 防自欺纪律):
  1. 逃逸用例必须被拒且事件流可见:写工作区外路径 / 外联网络 / 超时挂死;
  2. 真实任务 e2e:如「用 node 把 workspace 里的 csv 转成 md 对比表」全链路绿;
  3. 授权流:未授权联网被拒 → UI 请求 → 授权后重跑成功。
